# Example: `payflow-demo`

This example demonstrates how Turnkey can power stablecoin payment rails for a fintech company ("Payflow"). It shows end-to-end merchant wallet management, policy-gated USDC sweeps into an omnibus treasury, and how Turnkey's policy engine blocks unauthorized transfers — even from a compromised API key.

Built on top of patterns from [`sweeper`](../sweeper/) (transfer automation, `ethSendTransaction`, `pollTransactionStatus`), [`kitchen-sink`](../kitchen-sink/) (`createWallet`, `createApiOnlyUsers`, `createPolicy`), and [`trading-runner`](../trading-runner/) (ERC-20 policy condition patterns).

## Getting started

### 1/ Cloning the example

Make sure you have `Node.js` installed locally; we recommend using Node v18+.

```bash
$ git clone https://github.com/<your-fork>/sdk
$ cd sdk/
$ corepack enable  # Install `pnpm`
$ pnpm install -r  # Install dependencies
$ pnpm run build-all  # Compile source code
$ cd examples/payflow-demo/
```

### 2/ Setting up Turnkey

The first step is to set up your Turnkey organization and account. By following the [Quickstart](https://docs.turnkey.com/getting-started/quickstart) guide, you should have:

- A public/private API key pair for Turnkey
- An organization ID

Once you've gathered these values, add them to a new `.env.local` file. Notice that your private key should be securely managed and **_never_** be committed to git.

```bash
$ cp .env.local.example .env.local
```

Now open `.env.local` and add the missing environment variables:

- `API_PUBLIC_KEY` — admin API public key
- `API_PRIVATE_KEY` — admin API private key
- `BASE_URL` — defaults to `https://api.turnkey.com`
- `ORGANIZATION_ID` — your Turnkey org ID

The automation user credentials (`AUTOMATION_API_PUBLIC_KEY`, `AUTOMATION_API_PRIVATE_KEY`) will be generated by the setup script.

### 3/ Running setup

```bash
$ pnpm setup
```

The setup script provisions the full architecture inside your Turnkey org:

1. Creates a **merchant HD wallet** with 3 derived accounts (one per merchant)
2. Creates a **treasury wallet** (separate HD seed) with an omnibus account
3. Creates a **non-root automation user** with a generated P256 API key pair
4. Creates an **ALLOW policy** restricting the automation user to USDC `transfer()` calls to the treasury only

All generated identifiers (wallet IDs, addresses, user ID, policy ID) are saved to `payflow.generated.json`.

After setup completes, copy the printed automation credentials into `.env.local`:

```
AUTOMATION_API_PUBLIC_KEY="<printed value>"
AUTOMATION_API_PRIVATE_KEY="<printed value>"
```

### 4/ Funding a merchant account

Use the [Circle Faucet](https://faucet.circle.com/) to send Sepolia USDC to one of the merchant account addresses printed by setup. Also send a small amount of Sepolia ETH for gas from any [Sepolia faucet](https://cloud.google.com/application/web3/faucet/ethereum/sepolia).

### 5/ Running the demo

```bash
$ pnpm demo
```

The demo script authenticates as the non-root automation user (not the admin) and runs:

- **Positive path**: sweeps USDC from a funded merchant account to the treasury. Prints the transaction hash and Etherscan link.
- **Negative path 1**: attempts a USDC transfer to a non-treasury address — rejected by the policy engine.
- **Negative path 2**: attempts a non-USDC ERC-20 transfer to the treasury — also rejected.


## Architecture

```
┌─────────────────────────────────────────────┐
│              Turnkey Organization            │
│                                              │
│  ┌──────────────────────┐  ┌──────────────┐ │
│  │  Merchant Wallet      │  │  Treasury    │ │
│  │  (1 HD seed)          │  │  Wallet      │ │
│  │                       │  │  (separate   │ │
│  │  ├── Account 0 (M0)   │  │   HD seed)   │ │
│  │  ├── Account 1 (M1)   │  │              │ │
│  │  └── Account 2 (M2)   │  │  └── Omnibus │ │
│  └──────────────────────┘  └──────────────┘ │
│                                              │
│  ┌──────────────────────────────────────────┐│
│  │ Policy: ALLOW USDC transfer() → Treasury ││
│  └──────────────────────────────────────────┘│
│                                              │
│  ┌──────────────────────┐                    │
│  │ Automation User       │                    │
│  │ (non-root, API-only)  │                    │
│  └──────────────────────┘                    │
└─────────────────────────────────────────────┘
```

**Why this shape:**

- **Wallet Accounts, not Wallets, per merchant.** A Turnkey Wallet is one HD seed, capped at 100 per org. A Wallet Account is a derived address and is unlimited. One merchant = one derived account, so Payflow scales to thousands of merchants.
- **Separate treasury wallet.** The highest-value address must not share a seed with merchant deposit keys.
- **Non-root automation user.** Root credentials bypass the policy engine. The automation user is non-root so the policy actually gates signing.
- **Single ALLOW policy + implicit deny.** One policy governs all merchant accounts. Everything not matching is rejected by default — no DENY policies needed.

## Turnkey primitives used

- **Wallets** — `createWallet` for the merchant HD wallet and the treasury wallet
- **Wallet Accounts** — `createWalletAccounts` to derive per-merchant addresses
- **Users** — `createApiOnlyUsers` for the non-root automation user
- **Policies** — `createPolicy` with an `EFFECT_ALLOW` condition pinning token contract, function selector, and padded treasury recipient
- **Signing** — `ethSendTransaction` + `pollTransactionStatus` (reused from the `sweeper` example)

## Assumptions and simplifications

- **Sepolia testnet** — uses Sepolia USDC, not mainnet
- **Manual faucet funding** — merchant accounts must be funded via the [Circle Faucet](https://faucet.circle.com/) before running the demo
- **Manual sweep trigger** — the demo is a CLI command; production would use indexer webhooks or a polling worker
- **Automation user created programmatically** — `setup.ts` generates a P256 key pair via `@turnkey/crypto` and creates the user via `createApiOnlyUsers`
- **Flat org structure** — all wallets in one org for PoC simplicity; production would use sub-organizations per merchant for isolation
- **No gas sponsorship** — the funded merchant account needs Sepolia ETH for gas
